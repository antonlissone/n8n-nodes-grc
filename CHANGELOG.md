#### [1.0.7](https://github.com/antonlissone/n8n-nodes-grc/compare/1.0.6...1.0.7)

Resolves the four blocking issues raised in the n8n community-node verification review:

- **Auth refactor**: `Sai360GrcBasicApi` now uses n8n's `preAuthentication` + `authenticate` (generic) credential pattern. Session token is fetched via `/api/login`, cached on the credential record (DB-backed, shared across workflows), injected on every request via the `bwise-session` header, and refreshed automatically via n8n's built-in 401 retry. Removes all manual `httpRequest()` calls in credentialed paths and resolves the `no-http-request-with-manual-auth` lint violation.
- **Error handling**: HTTP failures (4xx/5xx) are now centrally converted to `NodeApiError` inside the transport layer. The error includes a structured payload (statusCode, body, headers), `httpCode` for the executions UI, and a description containing parsed SAI360 message arrays (FATAL/ERROR/WARNING/INFO/DEBUG/TRACE). For XML POST failures, `/api/log` is fetched automatically and appended to the description. Removes ~80 lines of duplicated `if (response.isError) throw new Error(...)` code from 9 resource files.
- **Dependency hygiene**: Removed the `overrides` block (eslint, rimraf, glob, @humanwhocodes/*) from `package.json` so the host environment's dependency tree is not constrained by build-tool concerns.
- **Documentation**: README rewritten with minimum tested n8n version (`2.18.4`), per-operation reference, two worked usage examples (datastore query and JSON record save), and a clear note that datastores/classes are customer-specific (no defaults shipped). API documentation reference points to each tenant's private `/console/docs`.
- **UX**: Sentence-case fix in the `Datastore identifier` parameter description.

##### Behaviour changes worth noting

- **Session caching scope** moved from per-workflow (`getWorkflowStaticData`) to per-credential (DB). Multiple workflows sharing one SAI360 credential now share one session — fewer `/api/login` calls.
- **HTTP 403** is no longer treated as session expiry. It now correctly surfaces as an authorisation error (previously it triggered a re-login + retry).
